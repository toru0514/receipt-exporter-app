# Receipt Exporter App 調査レポート

## 1. アプリ概要

Amazon の注文確認メールから経費情報を自動抽出し、Google スプレッドシートにエクスポートする Web アプリケーション。

**バージョン:** 0.1.0（MVP / 初期段階）
**コミット履歴:** 初期実装のみ（1コミット）

## 4. 主要機能と処理フロー

### ワークフロー（3ステップ）

1. **メール取得** - Gmail から Amazon 注文確認メールを取得（最大20件）
2. **AI 解析** - 選択したメールを Gemini 2.0 Flash で解析し、注文情報を構造化データとして抽出
3. **エクスポート** - 抽出結果を Google スプレッドシートに出力（新規作成 or 既存に追記）

### API エンドポイント

| エンドポイント | メソッド | 機能 |
|--------------|--------|------|
| `/api/auth/[...nextauth]` | GET/POST | Google OAuth 認証フロー |
| `/api/gmail` | GET | Amazon 注文メールの取得 |
| `/api/analyze` | POST | メール本文の AI 解析 |
| `/api/sheets` | POST | スプレッドシートへのエクスポート |

### 抽出データ項目

- 注文日、注文番号、商品名、金額、消費税、領収書リンク

## 5. 認証・認可

- **Google OAuth 2.0** を使用（NextAuth.js 経由）
- **要求スコープ:** openid, email, profile, gmail.readonly, spreadsheets
- **セッション管理:** JWT ベース（access_token, refresh_token 保持）
- **アクセスタイプ:** offline（リフレッシュトークン取得）

## 10. 現状の問題点・懸念事項

### セキュリティ

- API エンドポイントにレート制限なし（悪用リスク）
- 入力バリデーションが不十分（メール HTML をそのまま Gemini に送信）

### 信頼性

- API 呼び出しのリトライロジックなし（一時的エラーでそのまま失敗）
- Gemini のレスポンスから JSON を正規表現で抽出（脆弱なパース処理: `gemini.ts`）

- エラー発生時のリカバリ機構なし

- メール詳細取得が逐次処理（並列取得なし）
- メール解析も逐次実行（1件ずつ）
- ページネーション未実装（最大20件固定）

- AI 抽出データの検証・バリデーションなし（誤抽出の可能性）
- ページリフレッシュで結果が消失（永続化なし）
- 消費税の扱いがアイテム単位とオーダー単位で不整合の可能性

### 運用

- ログ基盤なし（`console.error` のみ）
- エラートラッキング・監視なし
- README がデフォルトテンプレートのまま（プロジェクト固有の説明なし）

### 未解決: Vercel本番環境でのGoogle OAuth Configuration エラー

#### 症状
- `https://receipt-exporter-app.vercel.app` でGoogleログインを試みると「アクセスをブロック: 認証エラー」
- NextAuth が `/api/auth/error?error=Configuration` にリダイレクトする
- 「There is a problem with the server configuration.」と表示される

#### 調査済み・排除した原因
1. **環境変数の未設定** → デバッグエンドポイントで全変数が正しく読み込まれていることを確認済み
   - `GOOGLE_CLIENT_ID`: あり（72文字、プレフィックス一致）
   - `GOOGLE_CLIENT_SECRET`: あり（35文字）
   - `AUTH_SECRET` / `NEXTAUTH_SECRET`: あり
   - `NEXTAUTH_URL`: `https://receipt-exporter-app.vercel.app`
   - `AUTH_TRUST_HOST`: あり
2. **環境変数の改行混入** → 最初 `echo` パイプが原因で `\n` が混入していたが、`printf` で全て再設定済み・確認済み
3. **`output: "standalone"`** → Vercelでは不要なのでコメントアウトしたが解消せず
4. **`trustHost: true`** → auth.ts に追加したが解消せず
5. **CSPヘッダー** → サーバーサイドのフェッチには影響しない
6. **GCPリダイレクトURI** → 正しく設定済み
7. **CSRF / プロバイダー設定** → `/api/auth/csrf` と `/api/auth/providers` は正常にレスポンスを返す

#### 次に試すこと（優先順）
1. **ミドルウェアの一時無効化** - `src/middleware.ts` が `/api/:path*` にマッチしており、`/api/auth/*` も対象。NextAuth v5 の内部処理と競合している可能性がある。`matcher` から `/api/auth` を除外してテスト
2. **Vercel Function Logs の確認** - Vercelダッシュボード > Deployments > 最新デプロイ > Function Logs でランタイムエラーの詳細を確認
3. **NextAuth v5 beta の既知の問題** - `next-auth@5.0.0-beta.30` はベータ版。Vercel本番環境固有のバグがある可能性。最新ベータに更新も検討
4. **Google OIDC Discovery** - NextAuth v5 の Google プロバイダーは OIDC を使用。サーバーサイドで `https://accounts.google.com/.well-known/openid-configuration` へのアクセスに問題がないか確認

### 補足事項
- Gemini API は無料枠（Free Tier）を使用中。`gemini-2.0-flash` はクォータ超過（`limit: 0`）で使用不可、`gemini-2.5-flash` を使用中
- Amazonのメールは別アカウントから手動転送しているため、`from:` ではなく `subject:fwd subject:発送済み` で検索
- 発送済みメールのみ金額情報を含むため、配達中・配達済みメールは検索対象外
- Vercel環境変数を `echo` でパイプすると改行が混入するので、必ず `printf` を使うこと
