# Receipt Exporter App 動作確認手順書

本ドキュメントは、Receipt Exporter App の各機能について動作確認を行うための手順書です。

---

## 1. 前提条件

### 1.1 事前準備の完了確認

以下の作業が「作業者実施内容.md」に従い全て完了していること。

- Google Cloud Console のプロジェクト作成済み
- Gmail API、Google Sheets API、Generative Language API（Gemini）が有効化済み
- OAuth 同意画面が構成済み（テストユーザーに自分のメールアドレスを追加済み）
- OAuth クライアント ID が発行済み（リダイレクト URI にローカル / 本番 URL を設定済み）
- Gemini API キーを取得済み（API キー方式を使う場合）
- `NEXTAUTH_SECRET` を `openssl rand -base64 32` で生成済み

### 1.2 必要な環境変数一覧と確認方法

`.env.local.example` をコピーして `.env.local` を作成し、以下の環境変数が全て設定されていることを確認する。

| 変数名 | 説明 | 確認方法 |
|--------|------|----------|
| `GOOGLE_CLIENT_ID` | Google OAuth クライアント ID | Google Cloud Console > 認証情報 で確認 |
| `GOOGLE_CLIENT_SECRET` | Google OAuth クライアントシークレット | Google Cloud Console > 認証情報 で確認 |
| `NEXTAUTH_URL` | アプリの URL（ローカルは `http://localhost:3000`） | `.env.local` の値を確認 |
| `NEXTAUTH_SECRET` | NextAuth セッション暗号化キー | `openssl rand -base64 32` で生成した32文字以上の文字列であること |
| `GOOGLE_GEMINI_API_KEY` | Gemini API キー（任意） | Google AI Studio で確認 |
| `CRON_SECRET` | Cron ジョブ認証用シークレット（任意） | 設定済みであること |

**確認コマンド**:

```bash
# .env.local ファイルの存在確認
ls -la .env.local

# 環境変数のキーが全て存在するか確認（値は表示しない）
grep -o '^[A-Z_]*=' .env.local
```

期待される出力:
```
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
NEXTAUTH_URL=
NEXTAUTH_SECRET=
GOOGLE_GEMINI_API_KEY=
CRON_SECRET=
```

### 1.3 依存パッケージのインストール

```bash
npm install
```

期待される結果: エラーなくインストールが完了すること。

## 2. ローカル環境での動作確認

### 2.1 起動確認

#### 手順

1. ターミナルで以下のコマンドを実行する

   ```bash
   npm run dev
   ```

2. ブラウザで `http://localhost:3000` にアクセスする

#### 期待される結果

- ターミナルに `Ready` の表示が出る
- ブラウザに「Amazon 経費管理」というタイトルと「Googleアカウントでログイン」ボタンが表示される
- コンソールにエラーが出ていないこと（ブラウザの DevTools > Console タブで確認）

#### 確認ポイント

- ページが正常にレンダリングされていること
- 「Amazonの注文確認メールから経費情報を自動抽出し、Google Sheetsに記録します」という説明文が表示されていること

> スクリーンショットポイント: ログイン前のトップページ全体

### 2.2 認証フロー確認

#### 2.2.1 Googleログイン

**手順**:

1. トップページの「Googleアカウントでログイン」ボタンをクリックする
2. Google のログイン画面が表示される
3. OAuth 同意画面でテストユーザーとして登録済みの Google アカウントを選択する
4. スコープの確認画面が表示された場合は「許可」をクリックする

**期待される結果**:

- Google の OAuth 同意画面が正常に表示されること
- ログイン後、アプリのメインページにリダイレクトされること
- ヘッダー右上にログインしたユーザーのメールアドレスが表示されること
- ヘッダーに「ログアウト」ボタンが表示されること
- 「1. Amazon メールを取得」セクションが表示されること

**確認ポイント**:

- リダイレクト URI が正しく設定されていること（`http://localhost:3000/api/auth/callback/google`）
- 「このアプリは Google で確認されていません」という警告が出た場合は「詳細」>「(アプリ名) に移動」で進められること（テストモードの場合）

> スクリーンショットポイント: OAuth 同意画面、ログイン後のメインページ

#### 2.2.2 セッション確認

**手順**:

1. ログイン後、ブラウザを新しいタブで開き `http://localhost:3000` にアクセスする

**期待される結果**:

- ヘッダーにメールアドレスと「ログアウト」ボタンが表示されること

#### 2.2.3 ログアウト

**手順**:

1. ヘッダー右上の「ログアウト」ボタンをクリックする

**期待される結果**:

### 2.3 メール取得確認

#### 前提

- Google ログイン済みであること
- テスト用の Google アカウントに Amazon の注文確認メールが存在すること

#### 2.3.1 メール取得ボタンの動作

**手順**:

1. 「1. Amazon メールを取得」セクションの「メールを取得」ボタンをクリックする

**期待される結果**:

- 画面上部に「メールを取得中...」というローディングメッセージ（青い帯）が表示されること
- メール取得中、スケルトンローダー（5行分のアニメーション付きプレースホルダー）が表示されること
- 取得完了後、Amazon の注文確認メール一覧が表示されること
- 各メールには以下の情報が表示されること
  - チェックボックス（デフォルトで全て選択済み）
  - 件名
  - 日付
  - スニペット（メール本文の抜粋）
- ヘッダー部分に「すべて選択 (N/N)」と表示されること（N はメール件数）

**確認ポイント**:

- 「メールを取得」ボタンがローディング中は無効化（disabled）されていること
- メールが最大20件まで取得されること（API のデフォルト取得件数）

> スクリーンショットポイント: スケルトンローダー表示中、メール一覧表示後

#### 2.3.2 メールが0件の場合

**手順**:

1. Amazon の注文確認メールが存在しない Google アカウントでログインする
2. 「メールを取得」ボタンをクリックする

**期待される結果**:

- 「Amazonからの注文確認メールが見つかりませんでした」というメッセージが表示されること

#### 2.3.3 日付フィルタの動作確認

**手順**:

1. フィルタパネル内の「開始日」に日付を入力する（例: 2025-01-01）
2. 「終了日」に日付を入力する（例: 2025-03-31）
3. 「適用」ボタンをクリックする
4. 「メールを取得」ボタンをクリックする

**期待される結果**:

- 指定した日付範囲内のメールのみが取得されること
- 「適用」ボタンは開始日または終了日が入力されるまで無効化されていること
- 日付フィルタ設定後、「クリア」ボタンが表示されること
- 「クリア」ボタンをクリックすると日付フィルタがリセットされること

#### 2.3.4 リージョン選択の動作確認

**手順**:

1. リージョンのドロップダウンを「Amazon.com」に変更する
2. 「メールを取得」ボタンをクリックする
3. リージョンを「すべて」に変更して再度取得する

**期待される結果**:

- 「Amazon.co.jp」: 日本の Amazon（amazon.co.jp）からのメールのみ取得される
- 「Amazon.com」: 米国の Amazon（amazon.com）からのメールのみ取得される
- 「すべて」: 両方のリージョンのメールが取得される

#### 2.3.5 メール選択の動作

**手順**:

1. メール一覧で個別のチェックボックスをクリックして選択/解除する
2. 「すべて選択」チェックボックスをクリックして全選択/全解除する

**期待される結果**:

- 個別チェックボックスの ON/OFF が正しく切り替わること
- 「すべて選択」チェックボックスで一括選択/解除ができること
- 選択件数の表示が正しく更新されること（例: 「すべて選択 (3/5)」）

### 2.4 AI解析確認

#### 前提

- メール取得が完了し、1件以上のメールが選択されていること
- `GOOGLE_GEMINI_API_KEY` が正しく設定されていること

#### 2.4.1 解析ボタンの動作

**手順**:

1. メール一覧で解析したいメールを選択する（チェックボックスで選択）
2. 「2. AI解析で注文情報を抽出」セクションの「選択したメールを解析 (N件)」ボタンをクリックする

**期待される結果**:

- 画面上部に「N件のメールを解析中...」というローディングメッセージが表示されること
- スケルトンローダー（テーブル形式のプレースホルダー）が表示されること
- 1件ずつ順番に解析が進行し、解析結果テーブルが徐々に更新されること
- 全件解析完了後、ローディングメッセージが消えること

**確認ポイント**:

- メールが未選択（0件）の場合、「メールを選択してください」というアラートが表示されること
- 「選択したメールを解析」ボタンに選択件数が表示されていること
- 解析中はボタンが無効化されること

> スクリーンショットポイント: スケルトンローダー表示中、解析進行中（一部結果表示済み）

#### 2.4.2 解析結果の表示確認

**手順**:

1. 解析完了後の結果テーブルを確認する

**期待される結果**:

テーブルに以下のカラムが表示されること:

| カラム | 表示内容の例 |
|--------|-------------|
| 注文日 | 2025/01/15 |
| 注文番号 | 250-1234567-8901234 |
| 商品名 | （商品の名前） |
| 金額 | ¥1,980 |
| 消費税 | ¥198 |
| 領収書 | 「開く」リンク |

**確認ポイント**:

- 金額が日本円のフォーマット（¥ + カンマ区切り）で表示されていること
- 領収書の「開く」リンクをクリックすると Amazon の領収書ページが新しいタブで開くこと
- 1つの注文に複数商品がある場合、各商品が別の行として表示されること

> スクリーンショットポイント: 解析結果テーブル全体

#### 2.4.3 部分的な失敗時の表示確認

**手順**:

1. 複数メールを選択して解析を実行する（一部が解析失敗するケースを想定）

**期待される結果**:

- 解析に失敗したメールがある場合、赤い帯で「N件のメールの解析に失敗しました」と表示されること
- 失敗したメールの件名とエラーメッセージが一覧表示されること
- 成功したメールの結果はテーブルに正常に表示されること

#### 2.4.4 検索フィルタの動作確認

**手順**:

1. 解析結果が表示された状態で、検索フィルタの入力欄に商品名の一部を入力する
2. 金額（数字）を入力する
3. 注文番号の一部を入力する
4. 「クリア」ボタンをクリックする

**期待される結果**:

- 商品名での検索: 入力した文字列を含む商品のみ表示されること
- 金額での検索: 入力した数値を含む金額の商品のみ表示されること
- 注文番号での検索: 入力した文字列を含む注文のみ表示されること
- リアルタイムにフィルタリングされること（入力のたびに即座に結果が変わる）
- 検索中は「N/M件の結果を表示中」と件数が表示されること
- 一致する結果がない場合は「検索条件に一致する結果がありません」と表示されること
- 「クリア」ボタンで検索がリセットされ、全結果が再表示されること

#### 2.4.5 バリデーション警告の表示確認

**手順**:

1. 解析結果が表示された状態で、EditableOrderTable コンポーネントが使われている場合にバリデーション結果を確認する

**期待される結果**:

- バリデーションサマリーが表示されること
  - エラーがある場合: 赤い帯に「N件のエラーがあります。セルをクリックして修正してください。」
  - 警告がある場合: 黄色い帯に「N件の警告があります。内容を確認してください。」
  - 問題なしの場合: 緑の帯に「全てのデータが正常です。」
- 問題のあるセルに赤（エラー）または黄色（警告）のドットが表示されること
- セルにマウスホバーするとツールチップで詳細が確認できること
- 「バリデーション詳細」をクリックすると全ての問題が一覧表示されること

#### 2.4.6 編集可能テーブルでの修正動作

**手順**:

1. 結果テーブルのセルをクリックする
2. 値を修正する
3. Enter キーまたはフォーカスを外して確定する
4. Escape キーでキャンセルする

**期待される結果**:

- セルをクリックすると入力フォームに変わること
- 値を修正して Enter を押すと新しい値が反映されること
- Escape を押すと修正がキャンセルされ元の値に戻ること
- 数値フィールド（金額、消費税、合計、数量）は数値入力のみ受け付けること
- 修正後にバリデーションが再実行されること
- 「データを確認」ボタンをクリックすると「確認済み」状態になること

**手順**:

1. Google Sheets へのエクスポートを1回以上実行した後、ページ下部の「エクスポート履歴」セクションを確認する
2. 「エクスポート履歴 (N件)」をクリックして展開する

**期待される結果**:

- エクスポート履歴がアコーディオン形式で表示されること
- 各履歴に以下の情報が表示されること
  - エクスポートしたデータ件数（例: 「5件のデータをエクスポート」）

- エクスポート履歴はブラウザの localStorage に保存されるため、ページリロード後も保持されること

curl -s http://localhost:3000/api/health | jq .

**期待される結果**:
